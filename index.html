Code Example: Form Validation

/* Style container when input is invalid and not empty */
.form-group:has(input:invalid:not(:placeholder-shown)) {
  border: 2px solid #ef4444;
  background-color: #fee2e2;

.form-group:has(input:valid:not(:placeholder-shown)) {
  border: 2px solid #10b981;
  background-color: #f0fdf4;
}

/* Style label based on input state */
.form-group:has(input:invalid) label {
  color: #ef4444;
  font-weight: 600;
}

/* Show error message only when invalid */
.form-group:has(input:invalid:not(:placeholder-shown)) .error-text {
  display: block;
}

Why Container Queries Beat Media Queries

Media queries respond to viewport width:

/* Media query - viewport-based */
@media (min-width: 768px) {
  .card {
    flex-direction: row;
  }
}

/* Problem: All cards change at same viewport width
   regardless of their actual container size */

Container queries respond to container width:

/* Container query - container-based */
@container (min-width: 20rem) {
  .card {
    flex-direction: row;
  }
}

/* Solution: Each card responds to its actual width
   Same component works in sidebar, grid, or main content */

Benefits:

  • ‚úÖ Component responds to container, not viewport
  • ‚úÖ True component reusability across contexts
  • ‚úÖ Works with flexible grids and dynamic layouts
  • ‚úÖ No need for context-specific CSS classes
  • ‚úÖ Components are self-aware of available space
  • ‚úÖ Perfect for component-based frameworks (React, Vue)

What is @starting-style?

@starting-style defines the initial state for entry animations. It works with display transitions to create smooth fade-in effects.

This enables animations that were previously impossible with pure CSS.

What is allow-discrete?

The transition-behavior: allow-discrete property enables transitions for discrete properties like display.

It makes the property wait until the end of the transition to change, making the animation visible.

Why is this useful?

Before these features, you couldn't animate elements appearing/disappearing with display:none.

You had to use visibility:hidden or opacity with workarounds. Now it's native CSS!

Code Example: Animating display

/* Element with display animation */
.box {
  display: block;
  opacity: 1;
  transition:
    opacity 0.4s ease-out,
    display 0.4s allow-discrete;
}

/* Hidden state */
.box.hidden {
  display: none;
  opacity: 0;
}

/* Starting state for entry animation */
@starting-style {
  .box {
    opacity: 0;
  }
}

Why This Matters

The old problem:

You couldn't animate display: none to display: block because CSS transitions don't work with discrete properties. Elements would just pop in/out.

The old workaround (with JavaScript):

// Had to use setTimeout and remove display manually
function fadeOut(element) {
  element.style.opacity = '0';
  setTimeout(() => {
    element.style.display = 'none';
  }, 300); // Match transition duration
}

function fadeIn(element) {
  element.style.display = 'block';
  element.style.opacity = '0';
  setTimeout(() => {
    element.style.opacity = '1';
  }, 10); // Force reflow
}

The new solution (pure CSS):

  • ‚úÖ @starting-style defines entry animation state
  • ‚úÖ allow-discrete enables display transitions
  • ‚úÖ No JavaScript timing hacks needed
  • ‚úÖ Smooth, declarative animations

Article Title: Understanding CSS scroll-state()

Container scroll-state queries allow you to style elements based on the scroll position and state of a scrollable container. This is particularly useful for implementing common UI patterns without JavaScript.

The scrollable Query

The scroll-state(scrollable: top) query returns true when the container CAN be scrolled upward ‚Äî meaning the user has scrolled down. Perfect for showing "back to top" buttons!

scroll-state(scrollable: bottom) returns true when there's more content below. Great for infinite scroll indicators or "more content" hints.

Why Use scrollable Over scrolled?

The scrollable query has much better browser support (Chrome 133+, Safari 18+) compared to scrolled which only works in Chrome 144+. For most use cases, scrollable is the right choice.

Implementation Pattern

Create a scrollable container with container-type: scroll-state and container-name. Then use container queries to style child elements based on scroll position.

The button below uses opacity: 0 by default and becomes visible when the container is scrollable to the top. Click it to smoothly scroll back up!

Progressive Enhancement

Always provide fallbacks for browsers without support. In this case, the button can remain visible or be hidden entirely depending on your design requirements.

Keep scrolling to see the button appear as you read through this article. The transition is smooth and controlled entirely by CSS ‚Äî no JavaScript scroll listeners required.

The stuck State

The scroll-state(stuck: top) query detects when a sticky element is actively stuck to the top of its scroll container. This is incredibly useful for adding visual feedback.

Common Use Cases

‚Ä¢ Add shadows to sticky headers when they stick
‚Ä¢ Change background colors for better contrast
‚Ä¢ Adjust padding or sizing of sticky elements

Without JavaScript

Before scroll-state(), detecting when a sticky element became stuck required Intersection Observer APIs or scroll event listeners. Now it's a simple CSS container query.

How It Works

The sticky element itself becomes the container with container-type: scroll-state. When it sticks, child elements can be styled using @container scroll-state(stuck: top).

JavaScript vs CSS

‚ùå Old Way: JavaScript

// Track scroll direction
let lastScrollTop = 0;
const header = document.querySelector('.header');

window.addEventListener('scroll', () => {
  const scrollTop = window.pageYOffset;

  if (scrollTop > lastScrollTop) {
    // Scrolling down
    header.style.transform = 'translateY(-100%)';
  } else {
    // Scrolling up
    header.style.transform = 'translateY(0)';
  }

  lastScrollTop = scrollTop;
}, { passive: true });

// Plus: debouncing, throttling,
// RAF scheduling, cleanup...

‚úÖ New Way: CSS

/* Container setup */
.page {
  container-name: page;
  container-type: scroll-state;
  overflow-y: auto;
}

/* Automatic scroll direction detection */
@container page scroll-state(scrolled: bottom) {
  .header { translate: 0 -100%; }
}

@container page scroll-state(scrolled: top) {
  .header { translate: 0 0; }
}

/* That's it! No JavaScript needed. */

Benefits of CSS scroll-state()

  • Zero JavaScript: Completely declarative, no event listeners or logic needed
  • Better Performance: Browser-native, runs on compositor thread
  • No Bugs: No scroll listener memory leaks or timing issues
  • Simpler Code: No state management, no throttling/debouncing
  • Accessible: Works with keyboard navigation and assistive technologies

üö® Important: Critical Implementation Rules

  • Cannot style container itself: Container queries only style descendants of the container, never the container element itself. Always wrap your content in a child element.
  • Must name containers: Use container-name (required) along with container-type: scroll-state.
  • Container must be scrollable or sticky: Add overflow: auto for scrollable containers, or position: sticky for stuck detection.
  • Use scrollTo() not anchor links: For back-to-top buttons in scrollable containers, use element.scrollTo({top: 0}) instead of <a href="#top"> which scrolls the entire page.
  • Browser support varies: scrollable/stuck work in Chrome 133+ and Safari 18+. scrolled requires Chrome 144+.
  • Value meanings: scrolled: top = scrolling UP (toward top), scrolled: bottom = scrolling DOWN (toward bottom). The value names the edge you're scrolling toward.
